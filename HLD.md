# Online Judge - High Level Design

## Overview:
An online judge platform allows users to:
• Register / Login
• View Problems
• Submit Code
• Get Verdicts (AC, WA etc.)

## Features:

### 1. User Management:
• Handles user registration and login (Authentication)
• Handles user roles like admin, normal user etc. (Authorization)
• Email verification for new registrations
• Password reset functionality
• User profile management

### 2. Problem Service:
• Handles CRUD on problems
• Problem categorization and search
• Test case management
• Difficulty level classification
• Problem statistics and analytics

### 3. Submission Service:
• Receives code and sends it to Evaluation Service
• Submission validation and preprocessing
• Queue management for evaluation
• Submission history tracking
• Real-time status updates

### 4. Evaluation Service:
• Evaluates the code and returns the verdict
• Multi-language code execution
• Performance monitoring (time, memory)
• Security enforcement
• Result validation and scoring

### 5. Real-time Communication:
• WebSocket-based live updates
• Real-time submission status
• Live user activity tracking
• System notifications

## High Level Design:

### 1. DB Design:

#### Collection 1 – Users:
```
{
  _id: ObjectId,
  username: String (unique),
  email: String (unique),
  password: String (hashed),
  type: Enum ["admin", "user"],
  hintUsage: Number
  isVerified: Boolean,
  verificationToken: String
  createdAt: Date,
  
}
```

#### Collection 2 – Problems:
```
{
  _id: ObjectId,
  title: String,
  description: String (markdown),
  difficulty: Enum ["easy", "medium", "hard"],
  createdAt: Date,
  updatedAt: Date,
}
```

#### Collection 3 – Testcases:
```
{
  _id: ObjectId,
  probemId: ObjectId (ref: Problems),
  input: String ,
  output: String,
  isSample: Boolean,
}
```

#### Collection 4 – Submissions:
```
{
  _id: ObjectId,
  problemId: ObjectId (ref: Problems),
  userId: ObjectId (ref: Users),
  code: String,
  language: Enum ["cpp", "java", "python"],
  verdict: Enum ["Pending", "AC", "WA", "TLE", "MLE", "CE", "RE"],
}
```

### 2. Web Server Design

#### UI Screens:

#### API Endpoints:

##### List Problems
- **Frontend**: Render React list displaying Problem titles and link them to individual problem page
- **Backend**: Define API for fetching all problems from the database using GET method
- **Endpoint**: `GET /api/v1/problems`

##### Individual Problem
- **Frontend**: Render Problem title and description along with other details like constraints and difficulty. Show a Coding area along with Run / Submit buttons
- **Backend**: Define API for fetching problem details through the id using GET method
- **Endpoint**: `GET /api/v1/problems/:id`

##### Code Submission
- **Frontend**: Render Submission Log after user submits the code for the problem
- **Backend**: Define API to handle POST request for code submission
- **Endpoint**: `POST /api/v1/submissions`
- **Request Body**:
  ```json
  {
    "problemId": "problem_id",
    "code": "user_code",
    "language": "cpp"
  }
  ```

After receiving the request, the following tasks will be executed:
1. Validate submission data
2. Store submission in database with "pending" status
3. Add submission to evaluation queue
4. Return submission ID to frontend
5. Evaluation Service picks submission from queue
6. Evaluate the code using a compiler / interpreter in the backend
7. Store the results generated by the code
8. Evaluate this result with the expected output for the corresponding input
9. Save the verdict for the submission
10. Send real-time update to frontend via WebSocket
11. Return the verdict to frontend where it will be shown in the submission log

### 3. Evaluation Service:

#### Docker Implementation:
• Since user can send malicious code into the system, it is important that we keep our servers secure
• Creating a docker container can enhance security
• We can create containers for specific compilers (eg: using Node image for JavaScript, GCC image for C++ etc.) for running codes of different languages
• A new container will be created each time a new code is submitted by the user. The code is evaluated inside the docker and then verdict is stored

#### Container Images:
- **C++**: `gcc:latest` - GCC compiler
- **Java**: `openjdk:latest` - OpenJDK runtime
- **Python**: `python:latest` - Python interpreter
- **JavaScript**: `node:latest` - Node.js runtime

#### Security Measures:
- **Resource Limits**: CPU, memory, and time restrictions
- **Network Isolation**: No external network access
- **Container Cleanup**: Automatic cleanup after execution

### 4. System Flow:

#### User Flow:
1. Register / Login
2. Browse Problems
3. Select Problem
4. Write Code
5. Submit Code
6. View Real-time Status
7. View Verdict and Results

#### Submission Flow:
1. User submits code
2. Code Submission Service validates and stores submission
3. Submission added to Code Submission Queue (Redis/Bull)
4. Evaluation Service picks submission from queue
5. Docker container created with appropriate language image
6. Code executed against all test cases
7. Results validated and scored
8. Verdict determined (AC, WA, TLE, etc.)
9. Results stored in database
10. Real-time update sent to frontend via WebSocket
11. User sees updated submission status

#### Real-time Communication Flow:
1. User connects to WebSocket service
2. User joins submission room
3. Evaluation Service processes submission
4. Results sent to WebSocket service
5. WebSocket service broadcasts to user's room
6. Frontend receives and displays update

### 6. Security:

#### Code Execution Security:
• **Containerised code execution** to prevent malicious code injection/ SQL injections and other code related security issues
• **Resource isolation** - CPU, memory, and time limits
• **Network isolation** - No external network access

#### Application Security:
• **Input sanitization** - Sanitising incoming input (example: sanitizing incoming problem description so it does not contain any malicious link)
• **JWT authentication** - Secure token-based authentication
• **Password hashing** - bcrypt with salt
• **CORS configuration** - Cross-origin security

#### Data Security:
• **MongoDB authentication** - Database access control
• **Environment variables** - Sensitive data protection
• **HTTPS** - Encrypted communication
• **Data encryption** - Sensitive data encryption at rest
